Metodologia MoSCoW:

--------------------------------------------------------------------------------
M1. REJESTRACJA I LOGOWANIE UŻYTKOWNIKÓW
--------------------------------------------------------------------------------
Status: ZAIMPLEMENTOWANE ✓

Komponenty:
• POST /api/auth/register - rejestracja nowego użytkownika
• POST /api/auth/login - logowanie, zwraca JWT token
• GET /api/auth/verify?token={token} - weryfikacja email

Uzasadnienie:
Bez systemu użytkowników nie ma możliwości identyfikacji sprzedawców,
przypisania aukcji ani zabezpieczenia danych. To fundament bezpieczeństwa.

Wymagania:
✓ Unikalne username i email
✓ Hashowanie haseł (BCrypt)
✓ Weryfikacja email przez token
✓ JWT dla bezstanowej autentykacji
✓ Role (USER, ADMIN)

Możliwe ryzyka:
• Ataki brute-force na hasła → rate limiting
• Spam rejestracje → CAPTCHA (przyszłość)

--------------------------------------------------------------------------------
M2. TWORZENIE I PRZEGLĄDANIE AUKCJI
--------------------------------------------------------------------------------
Status: ZAIMPLEMENTOWANE ✓

Komponenty:
• POST /api/auctions - utworzenie nowej aukcji
• GET /api/auctions - lista aktywnych aukcji (paginacja)
• GET /api/auctions/{id} - szczegóły aukcji

Uzasadnienie:
Główna funkcjonalność systemu - bez aukcji nie ma platformy do sprzedaży.
Przeglądanie ofert jest podstawową potrzebą kupujących.

Wymagania:
✓ Formularz z danymi pojazdu (VIN, marka, model, rok, przebieg...)
✓ Walidacja danych wejściowych
✓ Status ACTIVE dla nowych aukcji
✓ Automatyczne ustawianie expiresAt (+30 dni)
✓ Paginacja listy (wydajność)
✓ Publiczny dostęp do GET (bez logowania)

Metryki:
• ViewCount - liczba wyświetleń aukcji
• CreatedAt / UpdatedAt - timestamps

--------------------------------------------------------------------------------
M3. UPLOAD ZDJĘĆ DO AUKCJI
--------------------------------------------------------------------------------
Status: ZAIMPLEMENTOWANE ✓

Komponenty:
• POST /api/auctions/{id}/images - upload zdjęcia (multipart)

Uzasadnienie:
Zdjęcia są kluczowe dla sprzedaży samochodów. Kupujący muszą zobaczyć
stan pojazdu. Aukcje bez zdjęć są praktycznie niewidoczne.

Wymagania:
✓ Format: JPG, PNG, WebP
✓ Max rozmiar: 10 MB per file
✓ Unikalny naming: UUID_originalname.jpg
✓ Storage: katalog /uploads/
✓ Min. 1 zdjęcie, max. 15 zdjęć per aukcja
✓ Cascade delete - usunięcie aukcji usuwa zdjęcia

Zabezpieczenia:
✓ Walidacja typu MIME
✓ Sanityzacja nazwy pliku
✓ Sprawdzenie uprawnień (tylko owner)

--------------------------------------------------------------------------------
M4. EDYCJA I USUWANIE WŁASNYCH AUKCJI
--------------------------------------------------------------------------------
Status: ZAIMPLEMENTOWANE ✓

Komponenty:
• PUT /api/auctions/{id} - edycja aukcji
• DELETE /api/auctions/{id} - usunięcie aukcji

Uzasadnienie:
Sprzedawcy muszą móc poprawiać błędy, aktualizować cenę lub usunąć
aukcję gdy pojazd został sprzedany.

Wymagania:
✓ Autoryzacja - tylko owner może edytować
✓ Partial update (tylko podane pola)
✓ Walidacja biznesowa (np. cena >= 0)
✓ Cascade delete dla Car, Images

Zabezpieczenia:
• Sprawdzenie: auction.seller.id == currentUser.id
• 403 Forbidden jeśli nie owner

--------------------------------------------------------------------------------
M5. AUTORYZACJA I BEZPIECZEŃSTWO (JWT)
--------------------------------------------------------------------------------
Status: ZAIMPLEMENTOWANE ✓

Komponenty:
• JwtAuthenticationFilter - filtr na każdym requeście
• JwtService - generowanie i walidacja tokenów
• SecurityConfig - konfiguracja Spring Security

Uzasadnienie:
Bez zabezpieczeń system jest narażony na ataki, kradzież danych,
nieuprawnione modyfikacje.

Wymagania:
✓ JWT z podpisem cyfrowym (HMAC SHA-512)
✓ Expiration time (24h)
✓ Bearer token w headerze
✓ Autoryzacja na poziomie endpointów
✓ HTTPS only (produkcja)
✓ CORS configuration

Flow:
1. Login → generuje token
2. Każdy request: Authorization: Bearer {token}
3. Filter weryfikuje i ekstraktuje username
4. SecurityContext przechowuje Authentication

W1. SYSTEM PŁATNOŚCI ONLINE
--------------------------------------------------------------------------------
Powód pominięcia:
• Wymaga integracji z operatorami płatności (Stripe, PayU)
• Certyfikacja PCI DSS
• Obsługa zwrotów, reklamacji
• Koszty prowizji

Transakcje odbywają się poza platformą (gotówka, przelew).
Platforma jedynie łączy kupujących i sprzedających.

Przyszłość:
Można dodać jako opcję premium dla dealerów.

W2. SYSTEM AUKCJI "NA ŻYWO" (BIDDING)
--------------------------------------------------------------------------------

• Otomotus to platforma ogłoszeń, nie aukcji licytacyjnych

• Problemy z fake bids

Aktualnie:
Ustalona cena (lub negocjacje prywatne).

Przyszłość:
Osobna funkcjonalność "Licytacje" (opcjonalnie).
